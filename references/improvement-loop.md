# 개선 순환 설계 (순환 1: 자기교정 / 순환 2: 규칙 개선)

## 순환 1 — 실행 내 자기교정 (사람 개입 없음)

```
파싱 → [파싱 게이트] → 검사 → [검사 게이트] → 리포트
          ↑ 미달 시 보강(최대 2회)      ↑ 이상 시 재검(최대 2회)
```

**파싱 게이트 (2.5단계)** — `input-parsing.md` 체크리스트. 하나라도 "아니오/불확실"이면 보강(재렌더링·해상도 상향·표 재추출) 후 재점검. 2회 초과 미확보분은 "파싱 한계"로 리포트에 정직하게 기록.

**검사 게이트 (3.5단계)** — 아래 신호가 보이면 되돌아간다:

| 신호 | 조치 |
|---|---|
| `insufficient_spec` 비율 이상 고조 (요소의 30% 초과 기준) | 파싱 누락 의심 → 2.5단계로 복귀, 해당 요소 재확보 |
| Blocker인데 근거 빈약 | 목업/스펙 재확인 후 신뢰도 재분류 |
| 동종 요소 판정 불일치 | 규칙 적용 재점검 |
| 규칙표에 없는 지적 | "제안" 표시 → 순환 2 후보로 이관 |

되돌아가는 것은 정상 동작이다. 남은 불확실성은 숨기지 않고 "검사 한계" 섹션에 명시한다.

## 순환 2 — 규칙 개선 (사람 승인 필수)

**트리거**: 검토자가 "피드백 수집란"을 채운 리포트를 다시 스킬에 제시할 때만 작동. 매 검사 후 자동 실행 아님.

**절차**:
1. 리포트의 피드백란 + "규칙 개선 후보" 독취
2. 각 항목을 규칙 변경 제안으로 정리 — 대상 규칙 ID(신규/수정/삭제/심각도조정), 사유, 근거
3. `improvement-backlog.md`에 **추가 기록** (기존 내용 유지, 누적)
4. **master-rules.md는 스킬이 임의로 고치지 않는다** — 제안만 제시
5. 사람이 명시적으로 승인("이 제안 반영해")하면 그때 master-rules.md 수정 + 백로그에 변경 이력 기록

**안전장치**: 판정 기준의 변경 권한은 항상 사람에게 있다. 스킬은 근거 있는 제안 생성까지만.

## 백로그 기록 형식

`improvement-backlog.md`에 아래 형식으로 누적:

```
## [YYYY-MM-DD] 제안 #N — <한 줄 요약>
- 출처: <리포트 파일/피드백 항목>
- 유형: 신규 | 수정 | 삭제 | 심각도조정
- 대상: <규칙 ID 또는 "신규">
- 내용: <변경안>
- 근거: <오탐/미탐 사례, 원칙>
- 상태: 제안됨 | 승인됨(날짜) | 기각됨(사유)
```
